## 背景

当前 `GET /snapshot/{coinID}` 接口会读写内存缓存（`cache.Store`），但没有任何接口能够观测缓存的实时状态。运维人员和开发人员无法得知：当前缓存了哪些币种、各条目还有多久过期、哪些币种被频繁访问。本需求通过新增 `GET /cache/status` 接口来填补这一可观测性缺口。

---

## 功能需求

### FR-004-F1：新增 GET /cache/status 接口

系统须提供 `GET /cache/status` HTTP 接口，返回 HTTP 200，响应体为 JSON 对象。

**验收标准 AC-1**：当缓存为空时，接口返回 `{"coins": []}` 或等价的空数组结构，HTTP 状态码为 200。

**验收标准 AC-2**：当缓存中有至少一个未过期条目时，响应体中 `coins` 数组的长度等于当前未过期条目数量。

---

### FR-004-F2：每个条目包含币种标识

响应中每个币种条目须包含该币种的唯一标识符（与 `/snapshot/{coinID}` 中使用的 `coinID` 格式一致）。

**验收标准 AC-3**：若缓存中存有 `bitcoin` 和 `ethereum`，响应 `coins` 数组中须各有一个条目，其币种标识符分别为 `"bitcoin"` 和 `"ethereum"`。

---

### FR-004-F3：每个条目包含剩余 TTL

响应中每个币种条目须包含该条目的剩余生存时间，单位为秒，精确到整数（向下取整）。

**验收标准 AC-4**：若某条目在 45.7 秒后过期，其 TTL 字段值须为 `45`（整数，不含小数部分）。

**验收标准 AC-5**：不得返回 TTL 已耗尽（值 ≤ 0）的条目；若条目已过期，不得出现在响应中。

---

### FR-004-F4：每个条目包含命中次数

响应中每个币种条目须包含该条目自写入缓存以来被成功读取（缓存命中）的累计次数，类型为非负整数。

**验收标准 AC-6**：初次写入缓存时命中次数为 `0`。

**验收标准 AC-7**：每次通过 `GET /snapshot/{coinID}` 触发缓存命中，对应币种的命中次数递增 1。

**验收标准 AC-8**：缓存条目被 `Set` 覆写（同一 coinID 重新写入）后，命中次数重置为 `0`。

---

### FR-004-F5：响应体结构

响应体须符合以下 JSON Schema：

```json
{
  "coins": [
    {
      "coin_id":   "<string>",
      "ttl_sec":   "<integer, >= 0>",
      "hit_count": "<integer, >= 0>"
    }
  ]
}
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `coin_id` | string | 币种唯一标识，与 `/snapshot/{coinID}` 中的 coinID 一致 |
| `ttl_sec` | integer | 剩余 TTL，秒，向下取整，始终 >= 1 |
| `hit_count` | integer | 缓存命中累计次数，始终 >= 0 |

`coins` 数组的顺序不作规定（无序）。

**验收标准 AC-9**：响应 `Content-Type` 必须为 `application/json`。

## 非功能需求

### NFR-1：方法限制

`GET /cache/status` 仅接受 GET 请求。其他 HTTP 方法（POST、DELETE 等）的行为不在本需求范围内（由 Go 标准库 ServeMux 默认处理即可）。

### NFR-2：并发安全

接口实现须在并发请求场景下不出现数据竞争，读取缓存状态期间不得阻塞正常的 `GET /snapshot/{coinID}` 写入操作超过必要时间。

### NFR-3：性能

`GET /cache/status` 的 P99 响应时间须在缓存条目数 ≤ 1000 时低于 50ms（本地环境）。

### NFR-4：不计入 /metrics 统计

`GET /cache/status` 请求不计入 `/metrics` 接口的请求统计（与现有 `/metrics` 路由本身不被统计的原则一致）。

---

## 排除范围

以下内容**不在**本需求范围内：

1. **缓存操作接口**：本接口为只读状态查询，不支持通过 HTTP 接口删除或修改缓存条目。
2. **历史命中率**：仅记录自当前进程启动以来的累计命中次数，不提供时间窗口内的命中率计算。
3. **缓存大小限制或淘汰策略**：不引入 LRU、LFU 等淘汰机制。
4. **持久化**：命中次数仅保存在内存中，进程重启后归零。
5. **认证与鉴权**：本接口不引入访问控制机制。
6. **分页**：不对返回的币种列表进行分页处理。
